# Virro Enterprise Demo Platform — ISTQB Test Audit

**Execution date:** 2026-07-10  
**Test basis:** Virro master requirements, domain model, Analysis Packs, privacy principles and enterprise demo acceptance criteria.  
**System under test:** Next.js 16.2.10 application at commit `4197ef8` plus the security and QA hardening changes included with this report.  
**Environment:** Windows, Node.js 24.14.0, pnpm 11.7.0, Chromium in-app browser, production build on `localhost:3100`.  
**Viewports:** desktop 1440 × 900 and mobile 390 × 844.

## Executive verdict

**Conditional PASS for an enterprise demo. NO-GO for production use.**

The visual platform, mock analysis flow, demo scenarios, reports and responsive navigation meet the tested demo acceptance criteria. The production build is clean, the Prisma schema is valid, all 17 automated tests pass and the production dependency audit reports no known vulnerabilities.

Production release remains blocked until Virro implements authentication and authorization with workspace isolation, durable server-side persistence, and enforceable privacy/retention controls. Current scores remain correctly presented as estimates.

## Test strategy

The cycle follows the ISTQB test process:

1. **Planning and control:** identify critical business flows, security exposure and demo/production exit criteria.
2. **Analysis:** derive conditions from the master requirements and inspect the implementation statically.
3. **Design:** prioritize tests using risk-based testing; focus on Understanding Event creation, analysis consistency, executive outputs, sensitive-data handling and responsive operation.
4. **Implementation:** add unit, integration-style service and security configuration tests.
5. **Execution:** run static checks, production build, schema validation, dependency/secret checks and browser-based desktop/mobile journeys.
6. **Closure:** record defects, resolved findings, residual risks and release recommendation.

### Test levels and types

- Component: scoring, analyzers, sensitive-data redaction and domain helpers.
- Integration: Analysis Engine contract, demo/report services and cross-service score integrity.
- System: routes, inbox submission, scenario execution, reports and privacy UI.
- Non-functional: responsive layout, browser console health, HTTP security headers, unsafe method/path handling and dependency exposure.
- Static: ESLint, TypeScript production build, Prisma schema validation and tracked-secret review.

## Entry and exit criteria

### Entry criteria

- Application and seed data available locally.
- Requirements traceable to routes, domain types and services.
- Production build executable without external AI services.

### Demo exit criteria

- Lint, build, schema and automated test suite pass.
- Core journeys pass on desktop and mobile.
- No known production dependency vulnerabilities.
- No unhandled browser errors in tested journeys.
- Open limitations are documented and do not prevent a controlled demo.

### Production exit criteria

- All demo criteria pass.
- Authentication, authorization and tenant isolation verified.
- Understanding Events and privacy preferences persist server-side.
- Retention and deletion behavior is technically enforced and audited.
- Accessibility and security testing are expanded against the deployed environment.

The demo criteria pass. The production criteria do not.

## Security hardening applied

- Added a centralized sensitive-data sanitizer before analysis.
- Added security response headers and removed the framework disclosure header.
- Pinned direct dependencies and forced patched transitive `postcss` and `@hono/node-server` versions.
- Expanded ignore rules for environment files, database files, certificates, private keys and coverage artifacts.
- Added dedicated security tests and a reusable `test:security` command.
- Prevented a closed mobile sidebar from remaining interactive or exposed to assistive navigation.

## Residual risk and limitations

- The current CSP requires inline script/style allowances for this Next.js build. A nonce/hash strategy is recommended before production.
- There is no deployed API, identity boundary or data store in the tested demo; API authorization, CSRF and database security must be tested after those surfaces exist.
- No full WCAG 2.2 automated scan or complete keyboard/screen-reader audit was executed.
- Report export currently invokes the browser print flow; it does not create a managed PDF/CSV artifact.

## Release recommendation

Use this build for controlled sales and stakeholder demonstrations with mock data. Do not connect customer data or expose it as a production multi-tenant service until `VIR-001`, `VIR-002` and `VIR-003` in the defect catalog are resolved and independently retested.
